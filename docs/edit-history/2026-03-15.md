# 2026-03-15 — macOS cross-platform SAP GUI + auto-launch

## Summary

Added macOS support to the `sap-gui` library via a Scripting Console bridge (`_darwin.py`), refactored `session.py` into a platform dispatcher, and implemented auto-launch so SAP GUI for Java starts automatically on macOS. Multiple rounds of code review refined the implementation.

## Changes

### Cross-Platform Refactor

- **`session.py`** — Refactored from inline COM code to a platform dispatcher that imports `_win32` or `_darwin` based on `sys.platform`
- **`_win32.py`** — Extracted existing Windows COM code into its own module; added `auto_launch`/`connection_string` kwargs (warns and ignores on Windows) for API parity
- **`_darwin.py`** — New macOS backend using SAP GUI for Java's Scripting Console (Nashorn JS engine), AppleScript for UI automation, temp-file IPC for results

### macOS Scripting Console Bridge (`_darwin.py`)

- `_ScriptingBridge` — Pastes JS into the Scripting Console via AppleScript, clicks Play, reads result from temp file
- `_SAPElement` / `_SAPInfo` — Proxy objects providing the same API as Windows COM objects, converting PascalCase to camelCase
- IIFE wrapping prevents Nashorn scope collisions across consecutive runs
- `_execute_with_retry` handles console loss after screen transitions
- `_MIN_INTERVAL = 0.6s` throttle prevents clipboard/focus races
- `reset()` public method for resetting console state

### macOS Auto-Launch (`auto_launch=True`)

- `_read_landscape_connection()` — Parses `~/Library/Preferences/SAP/SAPGUILandscape.xml` for connection string
- `_detect_connection_string()` — Checks `SAP_CONNECTION` env var, falls back to landscape XML
- `_launch_sapgui()` — Launches SAP GUI for Java with `-o <conn_str> -b` flags
- `_kill_sapgui()` — Graceful kill with force-kill fallback
- `_poll_for_session()` — Probes bridge with short per-probe timeouts until session responds
- `connect()` handles 3 scenarios: (1) not running, (2) running but stale, (3) running with session

### Platform-Conditional Behaviour

- **`navigation.py`** — `run_transaction()` uses `startTransaction()` on macOS (bypasses modal popups), okcd + Enter on Windows
- **`export.py`** — `_fill_save_dialog()` reads default DY_PATH on macOS instead of setting it (Security Access Violation workaround), moves file via `shutil.move`
- **`ksb1/__init__.py`** — `_run_darwin()` batches entire KSB1 flow into a single JS bridge call (login separate); `run()` dispatches based on `sys.platform`

### KSB1 macOS Optimisation

- Single JS call: KSB1 navigate, clipboard cost centers (Java AWT), date fields, max hits, F8 execute, export menu, save dialog
- Cost center values validated with `isalnum()` to prevent JS injection
- Performance: ~38s total (was ~150s with individual bridge calls)

### Code Review Fixes (3 rounds)

- Added `_ScriptingBridge.reset()` public method (encapsulation)
- Updated `_execute_with_retry` to use `self.reset()` instead of direct `self._console_open = False`
- Added `auto_launch`/`connection_string` kwargs to Windows `SAPSession` for API parity
- Extracted `_LANDSCAPE_XML` to module-level constant
- Replaced fixed-iteration polling with `time.monotonic()` deadline in `_launch_sapgui()`
- Updated "3 bridge calls" comment to match reality ("login + single batched JS call")

- Warm reuse (SAP GUI already connected): PASSED
